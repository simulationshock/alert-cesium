# Implementation Plan: Multiplayer Collaborative Globe Sessions

**Branch**: `005-multiplayer-lobbies` | **Date**: 2026-05-15 | **Spec**: [spec.md](spec.md)

---

## Summary

Add real-time multiplayer to the CesiumJS globe app: OAuth authentication via Firebase Auth, ephemeral lobbies (max 6, public/private), WebRTC P2P mesh for voice/video/chat/map annotation sync, and Cesium presence avatars with pilot mode. A lightweight Node.js WebSocket signaling server handles only peer handshaking and lobby metadata — all application data flows P2P over data channels.

---

## Technical Context

**Language/Version**: TypeScript 5.x (browser `src/`), TypeScript 5.x compiled to Node.js (`server/`)
**Primary Dependencies**:
- Browser: Firebase Auth JS SDK, browser-native `RTCPeerConnection`, CesiumJS ^1.120
- Server: `ws` (WebSocket), `firebase-admin` (JWT validation), `dotenv`
**Storage**: In-memory only — no database. Ephemeral lobby state in Node.js process memory.
**Testing**: `tsc --noEmit` (type gate); Playwright e2e for auth flow and lobby join/leave
**Target Platform**: Browser (src/multiplayer/) + Node.js 20+ (server/)
**Project Type**: Browser library + companion signaling server
**Performance Goals**: Presence updates ≤200ms, annotation sync ≤1s, PTT audio ≤300ms
**Constraints**: No persistent storage, no SFU (v1), max 6 peers per lobby, signaling server carries zero application data
**Scale/Scope**: 6 users/lobby, arbitrary concurrent lobbies bounded by server RAM

## Constitution Check

**I. Browser-First Module Architecture**: `src/multiplayer/` contains only browser ESM — no `fs`, `path`, `http`, or Node.js-only APIs. The signaling server lives entirely in `server/` as a separate Node.js package that is never imported by `src/`. Firebase Admin SDK is `server/` only. Firebase Auth client SDK is browser-only. **PASS.**

**II. Immersive Comfort and Safety**: Pilot mode applies incoming camera state via per-frame `camera.setView()` with linear interpolation — identical to the existing `CameraFlight.ts` pattern. No abrupt jumps. When pilot mode ends (pilot leaves or follower opts out), camera control is restored immediately to the local user; no forced flight occurs. Viewer is always left in a valid interactive state. **PASS.**

**III. Explicit Outcome Contracts**: `MultiplayerSession.joinLobby()` returns a typed union: `{ type: 'connected' } | { type: 'lobby-full' } | { type: 'not-found' } | { type: 'invalid-code' } | { type: 'locked' } | { type: 'auth-failed' } | { type: 'network-error' }`. `followPeer()` returns `'following' | 'peer-not-found' | 'already-following'`. All async session mutations return typed outcomes — no silent state mutations. **PASS.**

**IV. Concern Isolation**: Multiplayer is a discrete concern in `src/multiplayer/`. It reads (but never writes) existing layer visibility state for layer sync. It does not modify `WildfireCameraMarkerManager`, `LiveFlightDataSource`, `EmergencyRadioMarkerManager`, or any existing module. The scope gate below explicitly rejects out-of-scope work.

**Scope gate — OUT OF SCOPE for this feature**:
- Modifying any existing data source or marker manager
- Camera flight behavior changes outside pilot mode
- Server-side annotation persistence or session recording
- SFU for larger lobby sizes
- Mobile-specific UI layout
- Guest (unauthenticated) lobby access

**PASS.**

**V. Build-First Validation**: `npm run build` (`tsc`) must pass at every task checkpoint. The signaling server has its own `tsconfig.json` and build step. Playwright e2e tests required for auth flow and lobby join/leave before merge. **PASS.**

**Gate status**: **PASS** — all five principles satisfied, no violations.

## Key Implementation Decisions

| Decision | Choice | Rationale |
|---|---|---|
| WebRTC library | Raw `RTCPeerConnection` | 6-peer mesh, browser-only; simple-peer adds overhead with no benefit |
| Auth provider | Firebase Auth | Unlimited free users, `signInWithPopup` works on GitHub Pages, Admin SDK for server token validation |
| Presence data channel | `ordered:false, maxRetransmits:0` | Drop stale frames rather than retransmit them |
| Annotation data channel | `ordered:true, maxRetransmits:3` | All annotations must arrive, separate channel prevents blocking |
| New joiner state sync | Temporary `transfer-state` channel from host | Avoids HOL blocking on annotation channel during bulk transfer |
| TURN credentials | Embedded in lobby-joined response | Single round-trip, atomic, ephemeral with 1h TTL |
| Pilot camera mirroring | Per-frame `camera.setView()` + lerp | Smooth, no overshoot, consistent with existing CameraFlight pattern |
| Sprite clustering | `billboard.pixelOffset` polar layout | Screen-space offsets, pilot centered, updated on `postUpdate` |
